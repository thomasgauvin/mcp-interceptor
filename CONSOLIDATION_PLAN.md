# MCP Interceptor Consolidation Plan

## Overview
This plan consolidates your two-worker monorepo (backend API + frontend) into a single unified Cloudflare Worker that serves both the REST API and React Router SSR frontend from one entry point, eliminating the need to run two independent workers.

**Current State:**
- Backend: Hono API with Durable Objects (5 REST endpoints, 2 WebSocket endpoints)
- Frontend: React Router 7 SSR application (2 routes)
- Deployment: 2 separate `npm run deploy` commands, 2 dev servers

**Final State:**
- Single Worker: Unified entry point serving API + frontend
- Single deployment: `npm run deploy`
- Single dev server: `npm run dev`
- No API host configuration needed (uses relative paths)

## 2. INTEGRATION: HONO API + REACT ROUTER IN ONE WORKER

### Architecture Pattern

The unified worker uses a **routing hierarchy**:

```
Request → src/index.ts → Route Decision Tree
  ├─ /api/* → Hono API app (all endpoints)
  ├─ /proxy/* → Hono API app (request forwarding)
  ├─ /monitor/* → Hono API (WebSocket upgrade)
  ├─ /static/* → Cloudflare static assets
  └─ /* → React Router SSR (everything else)
```

### Implementation Pattern (src/index.ts)

```typescript
import { Hono } from "hono";
import apiRoutes from "./api/routes";
import { createRequestHandler } from "react-router";
import MCPInterceptorDurableObject from "./durable-objects/mcp-interceptor";

export { MCPInterceptorDurableObject };

const app = new Hono<{ Bindings: Env }>();

// Mount Hono API routes with higher priority
app.route("/api", apiRoutes);
app.route("/proxy", apiRoutes);
app.route("/monitor", apiRoutes);

// Handle React Router for all other paths (fallback)
const reactRouterHandler = createRequestHandler(
  () => import("virtual:react-router/server-build"),
  import.meta.env.MODE
);

// Catch-all: React Router SSR
app.all("*", async (c) => {
  return await reactRouterHandler(c.req.raw, {
    cloudflare: { env: c.env, ctx: c.executionCtx },
  });
});

export default app;
```

**Why this works:**
- Hono's routing is evaluated first (exact path matches for API)
- React Router catches remaining requests as page routes or 404
- Durable Objects binding available to both API and frontend via `c.env`
- No manual request routing needed

### Critical Change: API Calls in Frontend

**Current approach (two workers):**
Frontend hardcodes backend domain in `.env`:
```
VITE_API_HOST="mcp-interceptor-api.appsinprogress.com"
VITE_HTTP_PROTOCOL="https"
```

**Unified approach:**
API calls become relative URLs (served from same origin):

```typescript
// Before (InterceptorCreator.tsx):
fetch(`${import.meta.env.VITE_HTTP_PROTOCOL}://${import.meta.env.VITE_API_HOST}/api/interceptors`, ...)

// After:
fetch(`/api/interceptors`, ...)
```

This eliminates the need for `VITE_API_HOST` and `VITE_HTTP_PROTOCOL` environment variables entirely.

## 3. BUILD/COMPILATION PIPELINE

### Build Process Flow

```
npm run build
  ├─ React Router build
  │   └─ Creates build/ directory with server build manifest
  │       ├─ build/server/index.js (SSR handler)
  │       └─ build/client/ (static assets)
  │
  └─ Vite + Wrangler compilation
      └─ Bundles src/index.ts + api/ + app/ code
          └─ Produces final worker bundle
```

### Updated vite.config.ts

```typescript
import { reactRouter } from "@react-router/dev/vite";
import { cloudflare } from "@cloudflare/vite-plugin";
import tailwindcss from "@tailwindcss/vite";
import { defineConfig } from "vite";
import tsconfigPaths from "vite-tsconfig-paths";

export default defineConfig({
  plugins: [
    cloudflare({
      viteEnvironment: { name: "ssr" },
    }),
    tailwindcss(),
    reactRouter({
      ssr: true,
    }),
    tsconfigPaths(),
  ],
  ssr: {
    resolve: {
      conditions: ["workerd", "browser"],
    },
  },
});
```

### Updated package.json Scripts

```json
{
  "scripts": {
    "dev": "wrangler dev",
    "build": "react-router build && wrangler deploy --dry-run",
    "deploy": "react-router build && wrangler deploy --minify",

    "cf-typegen": "wrangler types --env-interface CloudflareBindings",
    "postinstall": "npm run cf-typegen"

}
```

**Key differences:**
- `npm run build` runs `react-router build` first, then Vite compilation via wrangler
- Single `npm run deploy` replaces two separate deployments
- No separate backend/frontend build steps

## 4. CODE ORGANIZATION FOR MAINTAINABILITY

### API Layer Structure (src/api/)

**routes.ts** - Central API router:
```typescript
import { Hono } from "hono";
import { cors } from "hono/cors";
import * as interceptorHandlers from "./handlers/interceptor";
import * as proxyHandlers from "./handlers/proxy";
import * as monitorHandlers from "./handlers/monitor";
import * as validateHandlers from "./handlers/validate";

const api = new Hono<{ Bindings: Env }>();

// Enable CORS for API endpoints only
api.use("*", cors({ origin: "*" }));

// Interceptor management
api.post("/interceptors", interceptorHandlers.create);
api.get("/interceptors/:id", interceptorHandlers.get);
api.delete("/interceptors/:id/logs", interceptorHandlers.deleteLogs);

// Request proxying
api.all("/proxy/:id/*", proxyHandlers.forward);

// Monitoring
api.get("/monitor/:id", monitorHandlers.upgrade);

// Validation
api.post("/validate-mcp", validateHandlers.validateMcp);

export default api;
```

**handlers/** directory:
```
src/api/handlers/
├── interceptor.ts     (create, get, delete interceptors)
├── proxy.ts           (request forwarding)
├── monitor.ts         (WebSocket upgrade)
├── validate.ts        (MCP validation)
└── types.ts           (shared interfaces)
```

### App Layer (src/app/)

**entry.server.tsx** - Server rendering (no changes needed)

**routes.ts** - React Router config (no changes needed)

**components/** - All React components stay as-is (with API path updates)

**routes/** - Route files (home.tsx, monitor.tsx) stay as-is

## 9. REQUEST ROUTING (API vs STATIC vs PAGES)

### Request Routing Logic

```
GET /api/interceptors        → Hono handler (API)
POST /api/interceptors       → Hono handler (API)
GET /proxy/abc123/*          → Hono proxy handler
GET /monitor/abc123          → Hono WebSocket handler
GET /app.js                  → Static asset from build/client/
GET /styles.css              → Static asset from build/client/
GET /                        → React Router (home page)
GET /monitor/abc123          → React Router fallback (if no WebSocket header)
GET /any-path                → React Router (page or 404)
```

### Implementation in src/index.ts

```typescript
const app = new Hono<{ Bindings: Env }>();

// API routes (highest priority, exact matches)
app.route("/api", apiRoutes);
app.route("/proxy", apiRoutes);
app.route("/monitor", apiRoutes);

// Static assets (second priority, from build/client/)
if (c.req.path.match(/\.(js|css|png|jpg|svg|woff)$/)) {
  return await c.env.ASSETS.fetch(c.req.raw);
}

// React Router (lowest priority, catches everything else)
app.all("*", reactRouterHandler);
```

This ensures API requests are never intercepted by React Router.

## POTENTIAL CHALLENGES & MITIGATIONS

### Challenge 1: Worker Size Limit (1MB Bundled)

**Issue:** Combining Hono + React + build artifacts might exceed Cloudflare's 1MB limit.

**Expected size:** 800KB-950KB (typically under 1MB)

**Mitigations if size is an issue:**
- Use `--minify` flag (already in deploy script)
- Enable treeshaking in vite.config.ts
- Consider code splitting for React Router routes
- Monitor: `npm run build` shows bundle size

### Challenge 2: React Router Virtual Module Resolution

**Issue:** `virtual:react-router/server-build` must be resolved at runtime.

**Solution:** The `@cloudflare/vite-plugin` handles this automatically. Ensure plugin order in vite.config.ts is correct (cloudflare plugin first).

### Challenge 3: Environment Variable References in Frontend

**Issue:** Frontend code currently references `VITE_API_HOST` and `VITE_HTTP_PROTOCOL`.

**Required Changes:**
- Search for all `import.meta.env.VITE_API_HOST` references
- Replace with empty string (same origin) or remove completely
- Update all fetch URLs to relative paths:

```typescript
// Before:
`https://${import.meta.env.VITE_API_HOST}/api/interceptors`

// After:
`/api/interceptors`
```

### Challenge 4: CORS Configuration

**Current:** Backend enables CORS for all routes.

**Unified:** CORS only needed for `/api` routes (frontend is same-origin, no CORS needed).

```typescript
// In src/api/routes.ts
api.use("*", cors({ origin: "*" }));
// All API routes here
```

This prevents CORS headers from being added to static assets (improves performance).

### Challenge 5: Migration Path for Durable Objects Data

**Non-breaking:** Existing Durable Objects data persists automatically through consolidation.

The migration tag in wrangler.toml handles this:
```toml
[migrations]
tag = "v1"
new_sqlite_classes = ["MCPInterceptorDurableObject"]
```
